# Job-Seeker Platform — Senior Engineering Gap Audit
**Date:** 2026-07-08 · **Scope:** full codebase, logic/correctness focus · **Method:** core pipeline read directly + 3 independent subsystem audits (outreach, resume, email/analytics)

---

## The through-line (read this first)

This app was built as a **human-in-the-loop personal tool** and has since been silently converted to a **fully-automatic** system — but the safety gates, enforcement checks, and documentation were **not** updated to match. Two systemic consequences:

1. **The system now auto-sends AI-generated LinkedIn outreach and auto-tailored resumes to real people with no human review**, while the code comments, the README, and three separate routes still promise a review gate ("never blind-send") that **no longer exists in the code**.
2. **Several "limits" you configured are enforced only by asking the LLM politely in the prompt — not in code.** Your instinct was right: the salary floor *is* enforced deterministically, but the relevance threshold, seniority, and location rules are advisory-only.

The single highest-stakes individual bug is that the **resume "truthfulness gate" can be bypassed to fabricate metrics and skills** — i.e. it can get the user caught lying to a recruiter.

The whole app is **hard single-tenant** (this is the subject of the multi-user architecture phase).

## Category A — Limits/constraints NOT actually enforced (your core concern)

### A1 🔴 Relevance threshold is advisory-only
`relevanceThreshold` (65) appears in config, the settings UI, and *inside the LLM prompt text* — but **no code anywhere compares the returned score to it.** In `ai-scorer.ts:325-334`, a job is skipped only if the LLM itself populated `skipReason`. If the model returns `score:45, skipReason:null` (a common LLM inconsistency), the job passes as `NEW`. Confirmed by grep: the only `score <` in the tree is in the prompt string and a UI label.
**Impact:** compounds with A3/B1 — a misscored job gets auto-approved and auto-DMs a stranger.
**Fix:** after parsing, `if (parsed.score < threshold && !skipReason) { skipReason = "below_threshold"; skipCategory = "other"; }`

### A2 🟠 Seniority / location / role rejects are LLM-only past the title regex
`discover/route.ts` has a `TITLE_REJECT` regex, but it only inspects the **title**. JD-level seniority ("requires 5 years", generic title) relies entirely on the LLM rubric with no deterministic backstop.

### A3 🔴 FX outage silently disables the salary floor
`convertCurrency` (`fx.ts`) throws on a network failure; every caller does `.catch(() => null)`; `salaryGate(null, strict=false)` returns **pass** ("salary_unknown_kept"). A transient blip on the free `open.er-api.com` endpoint = every job's salary becomes "unknown" = the 14.5 LPA floor is not enforced for that whole run = mass auto-outreach. Same failure mode for any currency not in the FX table.
**Fix:** treat FX/normalization failure as "do not pass on salary" (or retry), not "unknown → keep".

### A4 🟠 Company-size filter is narrow and single-account
`filterByCompanySize` only works for jobs carrying a `companyId` (**LinkedIn only** — every other source passes through), fetches the profile using the **owner's** LinkedIn account, and passes through any unknown/failed size. So the "≥51 employees" rule covers a minority of jobs.

### A5 🟠 Settings PATCH has zero validation
`app/api/settings/route.ts` PATCH pipes `req.json()` straight into `updateSettings`. A malformed patch (`{search:{keywords:"backend"}}`) is persisted and merged; the next `discoverJobs()` does `keywords.map(...)` → **crash, whole discovery run dies**. Caps (`dailyInviteCap`, etc.) can be set to any value with no bounds.

### A6 🟢 Priority ranking uses config floor, not settings floor
`priority.ts` reads `config.search.minSalary.amount` while scoring reads `settings.search.minSalaryAmount` — board/digest ranking diverges from the live-tuned floor.

## Category B — The "never blind-send" promise is false (confirmed by 3 independent agents)

### B1 🔴 Auto-approve + auto-queue, no human gate
- `discover/route.ts:283-289`: every `NEW` job is immediately set `APPROVED` and `enqueueOutreach` is called inline.
- `enqueue.ts:126-137`: threads are created with `phase: QUEUED|CONNECTED` and `nextActionAt: new Date()` — the next in-window tick sends them.
- **But** the `enqueueOutreach` docstring (`enqueue.ts:6-9`) still says *"NOTHING SENDS YET … nextActionAt=null … the 'never blind-send' promise."* Grep confirms **nothing in the tree ever creates a `DRAFT`-phase thread.**
- **Dead code as a result:** `/api/outreach/confirm` always returns 409 (`phase !== "DRAFT"`), and `sweepStaleDrafts` (run every tick) matches zero rows forever.
- The README ("sends outreach after your approval") and the described HMAC-signed Approve/Skip email links are also stale — **that signed-link mechanism does not exist**; the digest just links to the dashboard.
**Fix:** decide the intended behavior and make code + docs consistent — either restore the DRAFT/`nextActionAt=null` gate, or delete the confirm route + draft sweep + all "never blind-send" claims. For a *commercial* product, an explicit review/opt-in gate is strongly advised.

## Category C — Resume truthfulness gate can be bypassed (highest individual stakes)

### C1 🔴 Whitelist is a subject-blind bag-of-tokens check
`whitelist.ts` accepts a replacement iff every claim-token in it appears **somewhere** in the master. There is no binding between a number and the achievement it quantifies, and no section awareness.
- **Fabricated metric:** master has "40 hrs/week"; LLM rewrites a bullet to "…cutting latency 40%". LaTeX `40\%` tokenizes to bare `40`, which exists anywhere → passes. A fake performance number ships on the resume the recruiter reads.
- **Recombined false claim (C2):** a "Coursework: Kubernetes" footer lets "operated Kubernetes clusters in production" pass — every token is in-vocab, the claim is invented.
- **Inflated YOE:** "5 years" passes whenever `5` appears anywhere; `years`/`experience` are in the generic-allow list.

### C2 🔴 The stated invariant ("never introduce a claim") ≠ what the code enforces ("no new token")
Any skill mentioned anywhere (coursework, a "familiar with" footer) can be recombined into a senior professional-experience bullet with zero new tokens.
**Fix (C1+C2):** constrain each edit so `replace`'s claim-tokens ⊆ its own `find` span's tokens (reorder/trim only — no importing tokens from elsewhere in the doc), and reject any number in `replace` not present in that edit's `find`.

### C3 🔴 Two-letter lowercase skills are invisible → freely inventable
`CLAIM_TOKEN`'s lowercase branch requires 3+ chars, so `go`, `ml`, `ai`, `ci`, `qa`, `db`, lowercase `js` are never tokenized or checked. "built microservices in go" passes with no Go anywhere in the master.

### C4 🔴 Transient compile outage permanently strands a job on the base resume
On an infra failure, `pipeline.ts` logs `tailorLog:{status:"failed"}`; the catch-up sweep only selects `tailorLog: null`, so **once any failure is logged the job is never auto-retried** — even after the compile service recovers. The cron documented as "resilience" won't retry the most common failure. Outreach then attaches the untailored base resume forever.

### C5 🟠 Other resume issues
- Manual tailored-PDF upload doesn't reconcile `altTailoredResumeKey` → the referral and direct-application resumes silently diverge (`api/resume/tailored/route.ts`).
- `PHONE_RE` matches any bare 10-digit run → can overwrite a 10-digit metric with the alt phone in the alt PDF (`alt-identity.ts`).
- No per-job tailor lock; discover's inline sweep and the hourly tailor cron use *different* locks → double LLM+compile on the same jobs + orphaned S3 objects (`pipeline.ts`).
- A legitimate repair that adds `\usepackage{...}` is rejected as "introduced new claims: fontspec" because `visibleText` keeps `{arg}` contents → job hard-fails on a correct fix.
- Edits validated against the pristine master but applied first-occurrence on the progressively-mutated doc → later edits can misfire/no-op silently.
- Anthropic path ignores `response_format` → weaker JSON reliability for a tailoring provider.
- The "sanity guard" is byte-size + page-count only (page count usually returns null) → does **not** catch wrong/mangled content as the docstring claims.

## Category D — Source fetching & salary correctness

### D1 🟠 Adzuna salary mishandled
`salary_min` is set to `amount/12` ("monthly") but Adzuna India figures are annual → the filter is mis-scaled. All Adzuna salaries are labeled `basis:"stated", confidence:"high"` even though many are Adzuna's own **predictions** (`salary_is_predicted` is ignored) → they bypass the estimate buffer in `salaryGate`.

### D2 🟠 JSearch currency defaults to USD
When `job_salary_currency` is absent, it defaults to `"USD"`. An INR figure (e.g. 800000) then gets treated as USD and converted to ~6.6 crore INR → sails over the floor. Salaries also labeled stated/high regardless of JSearch's aggregation.

### D3 🟠 Friend-digest salary display is wrong
`friend-digest.ts:14` computes `salaryMin/100000` assuming INR/year. A `$120k–$180k` role (stored raw in USD) renders **"1.2–1.8 LPA"** under the subject **"Job openings above 8 LPA"**. Eligibility uses the normalized `salaryAnnualBase` (correct) but the displayed range doesn't.

### D4 🟠 LinkedIn posts with unknown company/role collide and clobber
`toRawJob` falls back to `"Unknown (LinkedIn post)"` / `"Role from LinkedIn post"`. These generic strings produce an **identical `dedupeKey`**, so all-but-one unknown post is silently dropped; and the auto-DM to the post author reads *"I came across Unknown (LinkedIn post)'s Role from LinkedIn post opening."*

### D5 🟢 Lower-severity fetching gaps
- `dedupeKey` truncates role/company to 40 and city to 20 chars → distinct roles can collide and one is dropped.
- Blacklist matches substrings both ways (`co.includes(b) || b.includes(co)`) → "HCL" would block "veHCLe corp".
- Undated jobs (`postedAt` null) bypass the recency filter entirely.
- Ashby uses the short `descriptionSocial` blurb as the JD (worse scoring); Greenhouse uses `updated_at` as `postedAt` (edited-today looks fresh).
- **Unverified:** `date_posted: recencyDays` (=1) is passed to Unipile — if the API expects seconds or an enum, this filter silently no-ops and returns stale/all jobs. **Needs a live check.**
- ATS watchlist (`targetCompanies`) and `feedAuthors` are **empty** — the "highest-signal source" currently produces zero jobs.

## Category E — Outreach engine reliability

### E1 🔴 Distress-pause doesn't abort the batch
On a 429 / `account_restricted`, `handleSendError` trips `globalPause` but `processThread` just reschedules and returns — the `runOutreachTick` loop has no break and doesn't re-check the pause, so all remaining claimed threads still make one more live call to the just-restricted account. (`thread-worker.ts:275`, `outreach-tick.ts:116`)

### E2 🔴 Invite-timeout archive clobbers live threads
`doInvitePendingTimeout` re-fetches the profile; on a **transient** fetch failure it falls through to `cancelInvitation` + `archiveThread` unconditionally, and `archiveThread` only guards `status != ARCHIVED` (not REPLIED/CONNECTED). At the 14-day mark this can destroy a real accepted connection or overwrite a just-arrived REPLIED (webhooks hold no lock, so this also races). (`thread-worker.ts:389`)

### E3 🟠 Duplicate DMs on commit failure
`startChat`/`sendChatMessage` fire **before** `commitSend`. If the DB commit throws after the message left, state isn't advanced and the thread is reprocessed → a **second** DM. The `P2002` dedupe catch is inert: there is **no `@@unique([threadId, providerMessageId])`** in the schema, only an index. (`thread-worker.ts:98`)

### E4 🟠 Other outreach issues
- `pendingSendKey` is left set on error/already-connected/already-pending branches → thread stuck until the ~10-min reclaim sweep → 30–60 min stalls; circuit breaker slowed to a crawl.
- `reconcileInviteAcceptances` writes its "done for today" marker **before** doing the work → one transient API failure disables the missed-webhook safety net for the whole UTC day.
- A negative "please stop" reply sets `archivedReason` but not `status=ARCHIVED`/`archivedAt` → job shows as REPLIED (success-looking).
- Rate caps are **global**, not per-account (fine for the current single account; a blocker if accounts ever vary).

### E5 🟠 Analytics funnel inflated by unsent connections
1st-degree connection threads are created at `phase=CONNECTED, status=PENDING` **before** any DM. `aggregate.ts` classifies `phase===CONNECTED` as both **invitesSent** and **accepted** → a run that drafts 30 connection threads reports `invitesSent=30, accepted=30, 100%` before a single message goes out. The same seeding advances `appStage` to OUTREACH, inflating the pipeline too.

## Category F — Security / auth (single-tenant-acceptable now; blockers for commercial)

- **F1 🔴 (commercial):** one shared static password, default `"changeme"` if `APP_PASSWORD` unset, compared with `===` (timing-observable). All job mutations are gated only by this. `CRON_SECRET` compare is also non-constant-time.
- **F2 🟠:** `url-ingest` step 3 fetches an arbitrary user-supplied URL server-side with `redirect:follow` → **SSRF** (e.g. cloud metadata endpoints). Every pasted URL is also sent to `r.jina.ai` (third-party egress).
- **F3 🟠:** friend digest interpolates untrusted `company`/`role`/`jdText`/`applyUrl` into HTML and `href` **unescaped** → content/link injection into emails sent to third parties.
- **F4 🟢 (commercial):** no LLM spend cap anywhere → unbounded per-run/per-user cost.

## Category G — Staleness / state transitions

- **G1 🟠:** staleness sweep archives APPROVED jobs by outreach state only — it ignores `directAppliedAt` and `referredAt`, so a job the owner already applied to (direct) or got referred for is silently `SKIPPED` after 21 days, dropping a live application off the board.
- **G2 🟢:** `jobs/action` applies any `newStage` with no monotonicity guard and resets `approvedAt` on re-approve (restarts the staleness clock).
- **G3 🟢:** `test-friend-digest` with no `?to` emails the **real** friends with over-included jobs (any SKIPPED job with a salary, since `skipCategory` isn't persisted).

## The multi-user headline (feeds Phase 2)

Hard single-tenant in every layer:
- `AppSettings` and `ResumeProfile` are literally `id="default"` singleton rows.
- `Job`, `Contact`, `Outreach`, `ChannelThread`, `LlmUsage` have **no `userId`**.
- Rate-limit windows count **globally** across all threads; the LinkedIn account is `config.owner.linkedinAccountId`.
- Auth is a single shared password; there are no user accounts, sessions, or per-user isolation.
- The Unipile workspace, S3 bucket, SMTP sender, and AI provider keys are all single-owner.

## Things verified as CORRECT (so you don't re-check)

- Salary **gate** logic (confidence-scaled buffer over the floor) is sound and *is* enforced in code.
- Cross-run dedup (`dedupeJobs`) correctly blocks re-creating active/recent same-key/provider-id postings against the DB.
- Ticks and manual sends are serialized by `withCronLock("tick")` + a single-statement claim CTE — no cross-process double-send.
- Rolling 24h/7d rate windows + `min(daily, weekly)` budget: no calendar/timezone off-by-one; IST send-window math is correct.
- `enqueueOutreach` is genuinely idempotent (count check + per-job lock).
- Skip stops outreach (worker guards on `appStage==="SKIPPED"` before sending).
- Reply handling is idempotent across webhook + poll paths.
- Weekly-report double-send is correctly gated.
- Analytics division-by-zero is guarded.
- The DM never attaches the alt-identity resume (correct identity isolation).
